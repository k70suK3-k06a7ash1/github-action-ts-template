# ADR作成スキル

新しいADR（Architecture Decision Record）を作成します。

## ADR保存場所

`docs/decisions/`

## 命名規則

`{番号}-{slug}.md`

例: `010-new-feature-design.md`

## テンプレート

```markdown
# ADR-{番号}: {タイトル}

## ステータス

**Proposed** | **Accepted** | **Deprecated** | **Superseded by ADR-XXX**

## コンテキスト

この決定が必要になった背景や問題を説明。

## 決定

何を決定したかを明確に記述。

### 選択肢の比較（オプション）

| 選択肢 | メリット | デメリット |
|--------|----------|------------|
| A | ... | ... |
| B | ... | ... |

### 採用理由

なぜこの選択肢を採用したか。

## 結果

この決定による影響や結果。

### メリット

- ...

### デメリット・トレードオフ

- ...

### 今後の課題（オプション）

- ...
```

## 作成手順

1. 既存のADR番号を確認
   ```bash
   ls docs/decisions/
   ```

2. 次の番号でファイル作成

3. テンプレートに従って記述

4. README.md の更新（必要に応じて）

## 既存ADR一覧

| 番号 | タイトル |
|------|----------|
| 001 | アクションタイプ |
| 002 | バンドラー |
| 003 | テスト |
| 004 | Linting |
| 005 | Workflow DSL |
| 006 | パッケージマネージャー |
| 007 | プロジェクトスキャフォールディング |
| 008 | AI制御+関数型堅牢設計アーキテクチャ |
| 009 | XState・projen実装 |

## ベストプラクティス

- 1つの決定につき1つのADR
- コンテキストを十分に説明
- 却下した選択肢も記録
- 後から読んでも理解できるように
